# Security policy

## Supported code

Before the first public release, security fixes are applied to the current
development branch. After `1.0.0` is released, the latest `1.0.x` release will
be the supported public line unless a later policy states otherwise.

## Reporting a vulnerability

Please report vulnerabilities privately rather than opening a public issue.
Use a private GitHub security advisory if one is enabled for the eventual
public repository, or email all three corresponding authors:

- Haopeng Sun: `sunhaopeng@163.com`
- Yao Chen: `300630@njucm.edu.cn`
- Xin Xue: `xuexin@njucm.edu.cn`

Include the affected version or commit, impact, minimal reproduction steps,
and any proposed mitigation. Do not send real credentials, private service
URLs, proprietary files, licensed logs, patient information, or other personal
data. Use a harmless local fixture when demonstrating path or credential
handling.

The maintainers will coordinate validation, remediation, tests, and disclosure
with the reporter. Please allow time for a fix before public discussion.

Scientific disagreements, data corrections, and reproducibility failures that
do not create a security or privacy risk should be reported through the normal
issue workflow described in `CONTRIBUTING.md`.
