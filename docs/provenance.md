# Provenance and chain of custody

## Scope

This repository is a curated public snapshot, not a copy of the original working directory. It records the lineage needed to reproduce the released processed analysis for *Choosing among Competing Pockets in Oligomeric Proteins: An OIPS-Assisted, Traceable Multi-Evidence Analysis* while excluding local paths, service sessions, raw commercial formats, and unreviewed third-party packages.

Three records serve different purposes:

- `data/manifest.tsv` inventories every locally committed scientific asset with a repository-relative path or stable public URL, byte count, SHA-256, media type, access state, source version/date state, and license/terms statement.
- `data/SHA256SUMS` covers every file under `data/` except itself, including both manifests.
- `data/external_archive_manifest.tsv` records stable labels and availability/rights states for large or restricted asset classes that are not Git payloads. An entry is not evidence that the underlying artifact is locally available.

`data/metadata/asset-rights.tsv` is the redistribution gate, and `data/metadata/manual_decisions.tsv` is the scientific curation audit. Neither is replaced by narrative documentation.

## Curated source lineage

The deterministic curator is `scripts/prepare_public_data.py`. It selects only declared columns, normalizes types and missing values, applies stable sorting, and refuses overwrite without explicit authorization. The following source-export names identify logical project sources; they are not runtime dependencies of the released package.

| Released asset | Curated project source | Public transformation and stable order |
|---|---|---|
| `data/static/tool_pocket_features.csv` | `zhu-pack/pack/output/oips/oips_pocket_features.csv` | Select exactly the 14 public feature columns; normalize missing numeric values and types; sort by `pdb_id,tool,row_id`. The remaining source columns are not released or read by clustering. |
| `data/structures/*_paired.pdb` and `*_paired.cif` | `zhu-pack/pack/pdb_manifest.autogenerated.csv` plus approved paired structures | Read only `pdb_id` and the paired-structure locator during curation; copy approved files to stable lowercase names. Source job directories, entry names, and input-path fields are not released. |
| `data/metadata/systems.tsv` | `OIPS_FINAL_ANALYSIS/00_configuration/pocket_category_mapping.csv` and the structure manifest | Retain the approved target, system, oligomer, reference annotation, dataset-role, category, status, and family fields; add explicit structure/preparation/simulation metadata-status fields; sort by `pdb_id`. Unverified values are `not_available_in_source_materials`, not guesses. |
| `data/posthoc/reference_annotations.csv` | `zhu-pack/pack/output/oips/oips_target_summary.csv`, joined to reviewed system annotation and decision audit | Retain one target row with ligand key/status, released reference annotation, override flag, and decision ID; sort by `pdb_id`. |

The released input contains 1,742 records covering 21 targets. `row_id` is the stable raw-record identity. The feature table and prepared structures are the only scientific inputs to static clustering and ranking. The three `data/posthoc/` CSVs are loaded only after the static ranking has been serialized.

## Prepared structures and third-party origin

Each prepared structure is derived from an RCSB Protein Data Bank entry. `data/manifest.tsv` records the source PDB persistent identifier, local SHA-256, byte count, media type, and the RCSB terms statement. The repository distributes only structures marked `approved_for_repository` by the rights audit. Original structure authors and RCSB remain the source of the coordinate records; team preparation is a transformation, not a transfer of third-party ownership.

The public files do not establish the unavailable preparation details listed in `data/metadata/systems.tsv`, including assembly identifier, chain mapping, retained/deleted hetero groups, missing-atom handling, protonation, force field, water model, MD engine/settings, trajectory processing, or exact pocket-tool versions. Every target currently carries `metadata_status=author_confirmation_required` and `metadata_completeness=approved_category_fields_and_structure_provenance_only`. The committed prepared coordinates allow the released processed analysis to run, but they do not support reconstruction of every upstream preparation choice from raw archive coordinates.

## Manual decisions

`data/metadata/manual_decisions.tsv` supplies stable decision IDs, scope, type, field, prior and released values, rationale, evidence source, and approval status. The released audit includes:

- the 5LGE reference-key correction from `NAP:C:501:` to `6VN:D:503:`;
- the 21 target category annotations, including the three provisional four-class harmonizations;
- the frozen cluster-v2 parameter set; and
- selection of 5J89, 5TBM, and 4W9H as representative cases.

These decisions are data, not hidden conditionals. The code reads the released annotation/configuration and does not special-case 5LGE or representative cases in scientific functions.

## Rights and redistribution boundary

`data/metadata/asset-rights.tsv` separates three actions:

| Action | Meaning |
|---|---|
| `commit_local_payload` / `approved_for_repository` | The exact curated asset may be included, subject to its stated license scope and third-party terms. |
| `manifest_only` | Only team-authored inventory metadata may be included; the underlying asset is not redistributed. |
| `exclude` / `excluded` | Neither the asset nor identifying operational details belong in the public payload. |

Approved local payloads include the normalized feature table, minimal team-derived post-hoc summaries, reviewed metadata/audits, legacy crosswalk, and approved prepared structures. Manifest-only classes include full MD inputs and trajectories, restart/checkpoint files, representative MD snapshots, and large raw pocket-output packages. Licensed software binaries, logs, license details, session/job identifiers, result URLs, and unreviewed commercial formats are excluded.

Team authorization does not relicense RCSB coordinates, web-service outputs, or commercial-software-derived materials. Provider terms remain applicable. The external manifest currently uses explicit values such as `not_available_in_source_materials` and `not_archived`; blank persistent identifiers are not implicit DOIs.

## Computational lineage

The canonical output lineage is:

```text
data/static/tool_pocket_features.csv + prepared structures + config
  -> results/reference/clustering/*.csv
  -> results/reference/static/*.csv
  -> frozen static boundary
  -> data/posthoc/*.csv + systems metadata + prepared structures
  -> results/reference/analysis/*.csv
  -> results/reference/figure_source_data/*.csv
  -> figures/manuscript/*.svg
```

The five clustering tables retain candidate, membership, exhaustive source mapping, exclusion, and boundary-audit views. The two static tables contain only static fields. The 20 analysis tables map evidence and summarize metrics without altering those static tables. They include the ten one-at-a-time module-weight perturbations and the 14-target native single-tool complete-case comparison. The four repository-summary figure-source CSVs are rebuilt from validated analysis tables; figure code does not recompute scientific metrics. `MANUSCRIPT_RESULT_TRACEABILITY.tsv` separately maps each article figure/table to released data and states where final molecular artwork or unreleased trajectories prevent byte-for-byte figure regeneration.

## Run and release provenance

A normal full reproduction writes `run_manifest.json` at the reproduced bundle root. It records manifest schema, repository-relative configuration path and hash, random seed, consumed-input byte counts and SHA-256 values, produced-output byte counts and SHA-256 values, managed paths, frozen summary counts, and an environment fingerprint. The report builders render four derived reports:

- `rebuild_report.md`: configuration, inputs, outputs, managed paths, clustering counts, exclusions, maximum diameter, boundary flags, and hashes;
- `analysis_report.md`: validated analysis-table inventory, table fingerprints, headline metrics, evidence mappings, sensitivity outputs, and source hashes;
- `numeric_crosscheck.md`: actual, expected, delta, named tolerance, and pass/fail status for frozen numeric claims; and
- `validation_report.md`: aggregate check identifiers, statuses, summaries, and machine-readable evidence.

The frozen copies of these reports live under `results/reference/reports/`. Reference outputs are immutable: reproduction writes to `results/reproduced/`, and verification treats a path equal to or nested beneath `results/reference/` as compare-only.

`release/manifest.json` is the publication-level record. It is expected to bind the release version/tag and Git commit to configuration, input, environment, and key-result hashes. Until repository URL and archival identifiers are assigned, unavailable identifiers must be JSON `null` under an explicit `pre_publication_incomplete` status; they must not be fabricated.

## Validation chain

The release validation chain checks input JSON Schema contracts, exact headers, stable keys and sorts, data-manifest coverage, checksums, rights status, sensitive-content patterns, static/post-hoc directory separation, candidate→static and static→analysis field equality, one formal vote per tool, the 12.0 Å diameter cap, missing-aware score recomputation, frozen scientific snapshots, bootstrap metadata, `O_rel_formal` ablation, report consistency, and rebuilt figure-source equality.

CSV serialization is deterministic: UTF-8, LF, `%.15g`, empty numeric missing values, lowercase booleans, declared column order, and stable row order. Numeric claims use named tolerances rather than undocumented rounding. The scientific snapshot, not a prose value copied into a report, is the comparison source.
